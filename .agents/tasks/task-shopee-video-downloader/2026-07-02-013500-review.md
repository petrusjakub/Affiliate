# Shopee Video Downloader: security hardening pass (v2)

The four concerns from the v1 review — open SSRF proxy, disabled TLS verification, missing Content-Length cap, and absent server-handler tests — have all been addressed. The download proxy now validates target hostnames against a static allowlist plus regex patterns for dynamic Shopee CDN subdomains, enforces a configurable `MAX_DOWNLOAD_SIZE` (500 MB default), and uses `ssl.create_default_context()` without disabling verification. 18 new handler-level tests cover routing, path traversal, domain rejection, scheme rejection, oversized content, and the TLS context assertion. All 64 tests pass.

**Watch for:** The download proxy follows HTTP redirects after the domain check, so a hypothetical open-redirect on a Shopee CDN host could redirect to an internal IP (likely — depends on Shopee CDN behavior). The streaming size cap silently truncates the response mid-download when no Content-Length header is present, delivering a corrupt file without signaling the error to the client (confirmed).

**Verdict**: APPROVED

## High-level view

The SSRF surface is closed for direct requests: the allowlist correctly anchors patterns to Shopee TLDs and the test suite verifies that `attacker.shopee.co.id.evil.com` is rejected. The remaining gap is redirect-following — `urllib.request.urlopen` follows 3xx responses by default, so if an allowed CDN host issued a redirect to an internal IP the proxy would honour it. In practice Shopee CDN endpoints serve static media and don't issue open redirects, but this is a trust assumption rather than a code-level guarantee.

The download size enforcement has a clean pre-check path (Content-Length header present → 413 before streaming) and a streaming fallback (no header → silent truncation after the limit). The fallback path breaks the HTTP contract: the response headers are already sent, so the client receives a partial file with no indication it's incomplete.

## Redirect-following after domain allowlist check

The domain validation in `handle_download` runs once on the parsed URL before calling `urllib.request.urlopen`. Since `urlopen` uses the default `HTTPRedirectHandler`, a 3xx response from an allowed host redirecting to `http://169.254.169.254/...` would be followed transparently. The fix is straightforward: build an opener with a custom redirect handler that re-checks `is_allowed_download_domain` on each `Location` header, or use a `NoRedirectHandler` (similar to the one already in `resolve_short_url`) and reject non-CDN targets.

This is flagged as "likely" rather than "confirmed" because exploiting it requires an open-redirect vulnerability on a Shopee CDN endpoint, which is an external dependency outside this code's control. The code itself is provably redirect-following though — `urllib.request.urlopen` without a custom opener always follows redirects.

## Silent truncation in streaming fallback

```python
bytes_read += len(chunk)
if bytes_read > MAX_DOWNLOAD_SIZE:
    # Stop streaming if we exceed the limit even without
    # Content-Length header
    break
```

At this point the 200 status and headers are already on the wire. The client's download simply stops receiving data and may appear to succeed (if no Content-Length was promised) or fail with a size mismatch (if Content-Length was set but more data arrived than expected). In the no-Content-Length case, browsers may save the partial file without warning.

A better strategy: if the streaming limit is hit, close the underlying socket to force a client-visible error, or avoid committing headers until enough of the stream is buffered to know whether it'll exceed the cap. Neither is trivial with Python's `http.server`, so this is an acceptable trade-off for a personal-use tool — but worth documenting in a comment that the user may receive a truncated file.
