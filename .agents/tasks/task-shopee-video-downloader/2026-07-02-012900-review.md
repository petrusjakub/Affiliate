# Shopee Video Downloader: stdlib-only extraction proxy

A self-contained web application for downloading Shopee videos without watermarks, implemented entirely in Python's standard library. The server exposes two API endpoints — `/api/extract` (accepts a Shopee URL, returns video metadata) and `/api/download` (proxies the video stream to the browser). The frontend is vanilla HTML/CSS/JS with client-side URL validation mirroring the backend's regex patterns. Short URLs are resolved via a one-hop redirect follow, and direct video URLs are resolved against what appears to be `shopee.co.id/api/v4/video/get`.

**Watch for:** The download proxy is an open relay for any URL (confirmed). SSL certificate verification is disabled globally for all outbound requests (confirmed). The `video_id` is injected into the API URL without sanitization beyond regex extraction (likely low-risk but worth noting). The speculative API endpoint `shopee.co.id/api/v4/video/get` may not match Shopee's actual API surface (possible — untestable in this sandbox).

## High-level view

The download proxy endpoint accepts an arbitrary URL parameter and streams it through the server. Nothing constrains the target to Shopee CDN domains. An attacker can use the server as a proxy to reach internal network addresses or arbitrary external hosts — this is a server-side request forgery (SSRF) vector that should be gated to an allowlist.

Every outbound HTTPS connection (short-link resolution, API fetch, download proxy) sets `check_hostname = False` and `verify_mode = CERT_NONE`. This disables TLS trust entirely, making the server vulnerable to MITM interception of credentials or video content in transit. For a development-only tool this may be a deliberate trade-off, but it is undocumented and applies even to the download proxy path where user-requested content is streamed through.

The Shopee API endpoint and response structure are speculative — the code guesses several possible JSON shapes (`data.video_info`, `video_info`, top-level). If Shopee's actual API returns a different structure or requires authentication cookies/tokens, extraction will fail silently with a `VideoNotFoundError`. This is inherent to reverse-engineering an undocumented API and acceptable for a first pass, but worth documenting as a known fragility.

The test suite (46 tests, all passing) covers URL validation, response parsing, and the integration orchestration via mocks. No tests exercise the server's HTTP handling (path traversal defense, download proxy behavior, Content-Length handling).

<details>
<summary>Issues (4)</summary>

1. **Open download proxy (SSRF)** — `/api/download?url=` accepts any URL. Restrict to an allowlist of Shopee CDN domains (e.g., `*.shopee.co.id`, `*.shopee.sg`, CDN hostnames) before proxying.
2. **TLS verification disabled everywhere** — All `ssl.create_default_context()` calls immediately disable cert verification. At minimum, enable verification on the download proxy path so users aren't exposed to MITM'd content. Document the trade-off if disabling is intentional for Shopee API calls.
3. **No Content-Length cap on download proxy** — The proxy streams whatever the remote server sends with no upper bound. A malicious or unexpectedly large response could exhaust server memory or disk. Consider a configurable max response size.
4. **No server-handler tests** — The `ShopeeVideoHandler` class has no test coverage for its routing, path-traversal guards, or download proxy behavior. Adding at least a test for the path-traversal rejection and a mock-based proxy test would catch regressions.

## Open relay in the download proxy

The `/api/download` handler in `server.py` (line 205-252) takes a `url` query parameter, URL-decodes it, and opens a raw `urllib.request.urlopen` to that address with no domain restriction:

```python
url = urllib.parse.unquote(url)
# ...
response = urllib.request.urlopen(req, timeout=30, context=ctx)
```

The only consumer today is the frontend, which passes URLs returned by the extract endpoint. But the endpoint is publicly accessible. An attacker can request `GET /api/download?url=http://169.254.169.254/latest/meta-data/` and the server will proxy the response. The fix is straightforward: parse the URL before opening it and reject anything whose hostname doesn't match a set of known Shopee CDN patterns (e.g., `cf.shopee.co.id`, `cv.shopee.co.id`, `down-*.susercontent.com`).

## Disabled TLS verification

Three separate code paths create an SSL context and immediately neuter it:

- `resolve_short_url` (extractor.py line 120-121)
- `fetch_video_info` (extractor.py line 177-178)
- `handle_download` (server.py line 218-219)

All three do:
```python
ctx = ssl.create_default_context()
ctx.check_hostname = False
ctx.verify_mode = ssl.CERT_NONE
```

For the download proxy path, this is particularly concerning since the user is relying on the server to fetch genuine video content. A MITM could substitute arbitrary content and the server would deliver it without complaint.

## Speculative API contract and false-positive extraction

`fetch_video_info` constructs a request to `https://shopee.co.id/api/v4/video/get?video_id={id}` with headers mimicking a browser. `parse_video_response` speculatively searches for video URLs across several possible JSON structures (`video_url`, `url`, `play_url` fields; `formats`/`qualities` arrays).

The risk is that Shopee might return a 200 with an error payload that *contains* a URL-like string in an unexpected field (e.g., a redirect URL or ad URL), which would be treated as the video URL. There's no validation that the extracted URL actually points to video content before returning it to the user and piping it through the download proxy.

## Test coverage gaps

The 46 tests cover URL validation (14), video ID extraction (10), response parsing (10), `extract_video` orchestration with mocked network (7), server importability (3), and short URL resolution (2).

Not tested:
- The HTTP handler's routing logic, CORS headers, or error-to-status-code mapping
- Path traversal rejection in `serve_static`
- Download proxy behavior (streaming, error cases, the SSRF vector)
- Frontend JavaScript behavior (no JS test framework present)
